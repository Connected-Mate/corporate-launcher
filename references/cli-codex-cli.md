## 1. Overview

- **Vendor:** OpenAI.
- **Implementation:** Rust binary (not Node). Distributed via npm as a
  pre-built wrapper, plus standalone artifacts.
- **Install (user):** `npm i -g @openai/codex`
- **Install (corp):** the launcher pins a specific version and disables auto-update;
  the upstream binary lives wherever npm puts it (`$(npm prefix -g)/bin/codex`).
- **Official docs:** <https://developers.openai.com/codex>
- **Config root:** `$CODEX_HOME` (default `~/.codex/`).
- **Why it needs special handling:** Rust runtime means Node/Python env vars
  (`NODE_EXTRA_CA_CERTS`, `REQUESTS_CA_BUNDLE`) have no effect. The CLI also
  has incomplete `HTTPS_PROXY` support, and ships its own admin lockdown
  mechanism (`requirements.toml`) that no other CLI in this skill has.

## 2. Backend matrix

Codex CLI supports four backend shapes. Pick one during the interview
(`CX_BACKEND`) — every other variable cascades from this choice.

| Backend | `wire_api` | `base_url` shape | Auth env var |
|---|---|---|---|
| **Azure OpenAI** | `responses` | `https://<resource>.openai.azure.com/openai/v1` | `AZURE_OPENAI_API_KEY` |
| **OpenAI direct** | `responses` | `https://api.openai.com/v1` | `OPENAI_API_KEY` |
| **AWS Bedrock (gpt-oss models)** | `chat` | LiteLLM/Bedrock-gateway URL | `OPENAI_API_KEY` (passthrough token) |
| **Custom OpenAI-compatible** (LiteLLM, vLLM, on-prem) | `chat` | `https://litellm.internal/v1` | `OPENAI_API_KEY` |

Notes:

- Azure and OpenAI direct both use the **Responses API** (`wire_api = "responses"`).
- Everything else uses the **Chat Completions API** (`wire_api = "chat"`).
- Bedrock support is **only for OpenAI-published `gpt-oss-*` models** on Bedrock;
  there is no Anthropic-on-Bedrock path through Codex — that belongs to Claude Code.

## 3. Required env vars

The launcher exports these into the codex child process (see
`templates/codex-cli/launcher.sh.tpl`):

| Var | When |
|---|---|
| `OPENAI_API_KEY` | Always — fallback for tooling and MCP servers. |
| `AZURE_OPENAI_API_KEY` | Azure backend only. The launcher mirrors `OPENAI_API_KEY` into it. |
| `OPENAI_BASE_URL` | Always — exported for piggybacking tools, even when `wire_api = "responses"` uses the value from `config.toml`. |
| `CODEX_HOME` | Defaults to `~/.codex`; the launcher pins it to the install-managed dir if the tenant chose a custom location. |
| `CODEX_CA_CERTIFICATE` | Corp PEM bundle path (see §8). |

The provider-specific `env_key` declared in `config.toml` is what Codex
actually reads at runtime; the duplicate `OPENAI_API_KEY` export is a
compatibility belt for MCP servers and child processes.

## 4. `config.toml` structure

Path: `$CODEX_HOME/config.toml`. Generated by the launcher installer from
`templates/codex-cli/config.toml.tpl`.

Minimal shape:

```toml
model = "gpt-5-codex"
model_provider = "corp-gateway"
model_reasoning_effort = "medium"

disable_response_storage = true
hide_agent_reasoning = false

[history]
persistence = "none"

[telemetry]
enabled = false

[analytics]
enabled = false

[feedback]
enabled = false

[updates]
auto_check = false

approval_policy = "on-request"
sandbox_mode = "workspace-write"

[sandbox_workspace_write]
network_access = false
exclude_tmpdir_env_var = true
exclude_slash_tmp = true

[model_providers.corp-gateway]
name = "Corp gateway (azure)"
base_url = "https://my-resource.openai.azure.com/openai/v1"
env_key = "AZURE_OPENAI_API_KEY"
wire_api = "responses"
request_max_retries = 2
stream_idle_timeout_ms = 120000

[profiles.default]
model = "gpt-5-codex"
model_provider = "corp-gateway"

[profiles.fast]
model = "gpt-5.4"
model_provider = "corp-gateway"
model_reasoning_effort = "low"
```

Key rules:

- `model_provider` must match a `[model_providers.<id>]` table.
- `env_key` is the env var name Codex will read for the bearer token — pick
  it according to backend (Azure → `AZURE_OPENAI_API_KEY`, others → `OPENAI_API_KEY`).
- `[sandbox_workspace_write].network_access = false` is the corp default; the
  user can override only if the lockdown file (`requirements.toml`) allows it.

## 5. `requirements.toml` — admin lockdown

Path: `$CODEX_HOME/requirements.toml`, installed mode `0444` (read-only).
Generated only when `CX_REQUIRE_LOCKDOWN = "yes"` in the tenant config.
Source template: `templates/codex-cli/requirements.toml.tpl`.

This file is unique to Codex — no other CLI in the skill has a comparable
mechanism. If a value in the user's `config.toml` falls outside the
allowlists, **Codex refuses to start**.

Locked dimensions:

| Field | Effect |
|---|---|
| `allowed_model_providers = ["<id>"]` | Only the corp gateway provider can be used. |
| `forced_model_provider = "<id>"` | Override the user's choice. |
| `forced_login_method` | E.g. `"api_key"` — disables ChatGPT/OAuth personal login. |
| `allow_personal_api_login = false` | No personal OpenAI account. |
| `allowed_sandbox_modes = ["read-only", "workspace-write"]` | `danger-full-access` permanently excluded. |
| `forced_sandbox_mode = "workspace-write"` | The user's `config.toml` cannot soften this. |
| `allowed_approval_policies = ["untrusted", "on-failure", "on-request"]` | `never` not allowed. |
| `forced_approval_policy` | Tenant minimum. |
| `[sandbox_workspace_write] allow_network_access = false` | Sandbox is air-gapped from the user's network. |
| `allowed_mcp_servers = []` | See §9. |
| `[features] shell_tool_danger = false` | Blocks the unsandboxed shell variants. |
| `[features] auto_update = false` | Pins the version to whatever the launcher installed. |

The `[meta]` block at the end of the file is informational — it lets cyber
auditors see which tenant, authority and launcher version produced the file.

## 6. Azure OpenAI specifics

Azure is the most common corp backend and the most error-prone to configure.

```toml
[model_providers.azure]
name = "Azure OpenAI (corp)"
base_url = "https://<resource>.openai.azure.com/openai/v1"
env_key = "AZURE_OPENAI_API_KEY"
wire_api = "responses"
```

Critical rules:

- `base_url` MUST end with `/openai/v1`. **Do not** append `?api-version=...`
  — Codex constructs the path itself for the Responses API and a stray query
  string causes `400 Bad Request`.
- `wire_api` MUST be `"responses"`. Setting it to `"chat"` against an Azure
  Responses endpoint produces opaque `404`s.
- The Azure **deployment name** is what goes in `model` (e.g. `model = "gpt-5-codex"`
  only works if the Azure deployment is literally named `gpt-5-codex`). Mismatched
  deployment names are the #1 onboarding failure — surface a clear error in
  the launcher's `--status` output.
- The bearer is the Azure resource key, not an OpenAI key. The launcher
  exports it into `AZURE_OPENAI_API_KEY` and mirrors into `OPENAI_API_KEY`
  for MCP compatibility.

## 7. Known issues

### 7.1 `HTTPS_PROXY` partial support (upstream issue #4242)

Codex uses the Rust `reqwest` crate. As of May 2026 it honors `HTTPS_PROXY`
for the main HTTPS calls but ignores it for some auxiliary connections
(websocket, MCP transports, OCSP). On a strict corporate network where the
gateway is **not** directly reachable, this manifests as hangs or
`connection refused` errors.

Workarounds, in order of preference:

1. **Add the gateway hostname to `NO_PROXY`** so Codex talks to it directly.
   This is the default the launcher applies.
2. **Run a transparent proxy at the network layer** (Squid in intercept mode,
   nftables redirect). The CLI sees a normal TCP socket and never knows
   a proxy exists.
3. **Bind the gateway via VPN split-tunnel** so it lives on the routable
   side of the VPN, bypassing the corporate egress proxy entirely.

The launcher warns the user when both `HTTPS_PROXY` is set and
`CX_PROXY_WARNING = "yes"` — see `warn_proxy_quirk()` in `launcher.sh.tpl`.

### 7.2 No `NODE_EXTRA_CA_CERTS` honoring

Codex is Rust — it does not read Node env vars. Setting
`NODE_EXTRA_CA_CERTS` does nothing. Use `CODEX_CA_CERTIFICATE` (see §8).

### 7.3 Auto-update fights pin-management

Codex tries to self-update on launch by default. The lockdown file forces
`auto_update = false`, and the launcher additionally exports
`CODEX_DISABLE_UPDATE_CHECK=1`. Belt and braces.

## 8. Corporate CA bundle

Codex CLI uses the **OS-native trust store** plus, optionally, a path provided
via `CODEX_CA_CERTIFICATE`. It does **not** read `NODE_EXTRA_CA_CERTS`,
`REQUESTS_CA_BUNDLE`, or `SSL_CERT_FILE`.

```bash
export CODEX_CA_CERTIFICATE=/etc/ssl/acme-corp-bundle.pem
```

Caveats:

- The bundle file must be a single concatenated PEM — Codex will not walk a
  directory.
- On macOS, the bundle is **added to** (not replacing) the System Keychain
  roots. On Linux, the OS store is whatever `rustls-native-certs` finds in
  `/etc/ssl/certs/`.
- If TLS inspection is in use and the corp root CA is already in the OS
  store, you may leave `CODEX_CA_CERTIFICATE` unset — the launcher does this
  when `CA_BUNDLE_PATH` is empty.
- Never set `NODE_TLS_REJECT_UNAUTHORIZED=0` here — it has no effect (Codex
  is Rust) and is a security anti-pattern flagged by cyber.

## 9. MCP allowlist

Codex CLI supports MCP servers via `[mcp_servers.<id>]` in `config.toml`.
Because MCP servers run arbitrary local processes with the user's
permissions, the corporate baseline forbids ad-hoc additions.

Enforcement chain:

1. `requirements.toml` declares `allowed_mcp_servers = []` (empty by default).
2. Any `[mcp_servers.*]` block in the user's `config.toml` that does not match
   an entry in `allowed_mcp_servers` is rejected at startup.
3. The user cannot edit `requirements.toml` (mode `0444`).
4. Adding an MCP server requires re-running the installer with the new
   server pre-approved by `${CYBER_AUTHORITY}`.

Common asks (and the correct disposition):

| Server | Disposition |
|---|---|
| `filesystem` (official) | Approve case-by-case; restrict roots in args. |
| `git` | Usually safe; approve. |
| `puppeteer` / `playwright` | Reject — opens the browser to arbitrary URLs. |
| `shell` / unrestricted exec | Reject — defeats the sandbox. |
| Cloud connectors with persistent OAuth | Reject — leaks credentials into the CLI session. |

## 10. Model IDs (May 2026)

| Model | Use | Approx. pricing (per 1M tokens, input / output) |
|---|---|---|
| `gpt-5-codex` | Default coding model. Best quality on long-horizon agentic tasks. | $15 / $60 |
| `gpt-5.5` | General reasoning, faster than gpt-5. Use as `[profiles.balanced]`. | $5 / $20 |
| `gpt-5.4` | Cheap & fast — `[profiles.fast]` for commit messages, name suggestions. | $1.20 / $4.80 |

(Prices are the public OpenAI list rates as of May 2026; corp Azure deployments
are negotiated separately and the launcher does not surface them.)

Reasoning-effort settings (`model_reasoning_effort`):

- `low` — fast, single-pass.
- `medium` — default. Good trade-off for codex on real codebases.
- `high` — slow, deep planning. Avoid in `--status`/CI flows.

The launcher exposes a `[profiles.fast]` preset wired to `gpt-5.4` + `low`
so users can `codex -p fast "rename foo"` without round-tripping the pricey
default.

## 11. Telemetry kill

Layered defense — env vars in the launcher PLUS pinned values in `config.toml`
PLUS hard-locked values in `requirements.toml`.

**Launcher env vars** (from `disable_telemetry()` in `launcher.sh.tpl`):

```bash
export DO_NOT_TRACK=1
export DISABLE_TELEMETRY=1
export DISABLE_ERROR_REPORTING=1
export OTEL_EXPORTER_OTLP_ENDPOINT=""
export OTEL_SDK_DISABLED=true
export SENTRY_DSN=""
export DD_TRACE_ENABLED=0
export STATSIG_DISABLED=1
export CODEX_DISABLE_TELEMETRY=1
export CODEX_DISABLE_ANALYTICS=1
export CODEX_DISABLE_FEEDBACK=1
export CODEX_DISABLE_UPDATE_CHECK=1
```

**Pinned in `config.toml`:**

```toml
[telemetry]
enabled = false

[analytics]
enabled = false

[feedback]
enabled = false

[history]
persistence = "none"

[updates]
auto_check = false

disable_response_storage = true
```

**Hard-locked in `requirements.toml`:**

```toml
[telemetry]
allowed = false

[analytics]
allowed = false

[feedback]
allowed = false

[features]
external_telemetry = false
response_storage = false
history_persistence = false
auto_update = false
```

`disable_response_storage = true` is the most important line for tenants
with data-residency rules — it instructs Codex not to upload conversations
to the vendor for "improving the service".
